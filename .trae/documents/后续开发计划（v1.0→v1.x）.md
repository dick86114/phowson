## 当前基线（已完成）
- 数据层：已接入 Postgres 并实现云端持久化（Fastify + pg），前端已通过 HTTP API 访问数据。
- 功能：照片列表/详情、上传、后台管理、点赞/评论、统计接口已打通。
- 风险点：仍是“模拟登录”，权限控制只在前端 UI 层；图片存储仍以 DB bytea/外链跳转为主（后续需对象存储）。

## 总体目标（对齐 PRD）
- 高保真沉浸式画廊 + 高性能加载（瀑布流、虚拟列表、占位/渐进加载）。
- 上传工作流：EXIF 100% 读取 + AI 仅负责标题/描述/标签 +（可选）AI 摄影导师。
- 习惯养成：真实 Streak、热力图、挑战与成就。
- 多角色权限：Admin/Family/Guest 的后端 RBAC + 审计/安全策略。
- 分享与移动端：分享卡片、PWA、手势导航。

## 开发原则
- 先把“真实数据与权限边界”做扎实，再做体验类增强。
- 所有新功能以“可观测 + 可回滚”为前提：埋点/日志、开关（feature flag）、灰度。
- 前端数据流统一使用 TanStack Query；避免散落的 useEffect。

## Phase 0：稳定化与工程化（强烈建议先做）
- **后端结构化**：路由拆分（photos/users/stats/admin）、统一错误返回格式、请求校验（schema）。
- **鉴权占位升级**：把 `x-user-*` 头升级为真正 token（下一阶段接 Clerk/NextAuth），并在后端做 RBAC（至少：Guest 禁止写、Family/Guest 禁止后台）。
- **可观测性**：请求日志脱敏、慢查询日志、DB 连接池指标、统一 request-id。
- **测试**：加 API 冒烟测试（列表/详情/点赞/评论/上传/删除）。

交付物
- 后端模块化目录、通用错误处理中间件、最小测试集。

## Phase 1：身份认证与 RBAC（从“能用”到“可控”）
- **接入真实认证**：
  - 方案 A（推荐）：Clerk（前端拿 JWT，后端校验）
  - 方案 B：自建（邮箱+密码+JWT+刷新 token）
- **角色模型落库**：users.role 统一为 `admin|family|guest`（现在前端存在 ADMIN/GUEST 与小写混用，需要统一）。
- **后端权限矩阵**：
  - Admin：全量 CRUD、可见 AI 点评/统计
  - Family：可浏览/评论（按相册/可见性策略）
  - Guest：只读 + 点赞/评论（如需限制可配置）
- **后台改造**：把“用户管理/分类管理”的 UI 从纯前端临时状态，改成后端真实表（users/categories）。

交付物
- 真实登录、后端 RBAC、用户管理可用。

## Phase 2：对象存储与图片管线（解决性能与成本）
- **对象存储接入**：
  - 方案 A：Supabase Storage
  - 方案 B：S3/R2 + CDN
- **图片处理**：上传原图 + 生成多尺寸（thumb/medium/original），前端按场景请求。
- **数据库调整**：photos 表从 image_bytes 为主改为 `image_url + variants(jsonb)` 为主；保留 bytea 作为兼容/回滚。
- **安全**：上传鉴权、MIME/扩展名白名单、最大尺寸限制、图片病毒扫描（可选）。

交付物
- 可大规模存储与分发的图片体系，首页与详情加载明显提速。

## Phase 3：画廊体验升级（瀑布流 + 高性能加载）
- **瀑布流布局**：优先选择成熟库（若不引入库则实现测量 + columns 方案升级），支持不同宽高比。
- **虚拟列表/窗口化**：支持千级照片不掉帧。
- **占位与渐进加载**：Skeleton/BlurHash/低清预览（LQIP）。
- **交互细节**：全屏模式更稳定，键盘/手势、图片预加载下一张。

交付物
- Home/Detail 体验达到 PRD 的“沉浸+流畅”。

## Phase 4：上传工作流增强（批量 + EXIF 完整）
- **批量上传队列**：拖拽多文件、并发控制、失败重试、断点续传（可选）。
- **EXIF 统一规范**：
  - 前端使用 exifr 读取
  - 后端再做二次校验/归一（字段名统一），并存 JSONB
- **表单体验**：自动建议分类/标签；同批次复用设置。

交付物
- 摄影师“高效整理”闭环可用。

## Phase 5：AI 增强（填单 + 摄影导师）
- **Prompt 分层**：
  - 标题/描述/标签（对所有可用）
  - AI 点评（仅 Admin 可见，写入 photos.ai_critique）
- **成本控制**：缓存、批量时合并请求、失败降级。
- **安全**：不在前端暴露 key（把 Gemini 调用搬到后端或 serverless）。

交付物
- AI 真正融入上传流程，且可控可监控。

## Phase 6：习惯养成系统（Streak / Heatmap / 成就）
- **数据模型**：新增 activity_logs（按天记录上传/发布），streaks 或用 SQL 计算。
- **Heatmap 组件**：展示过去一年贡献，支持筛选（仅上传/仅发布/按分类）。
- **徽章/挑战**：规则引擎（简单版用 SQL + cron），每周命题可先静态再 AI 生成。

交付物
- 从“展示作品”升级为“养成系统”。

## Phase 7：地图模式与探索（Map View）
- **地理数据**：exif.gps（lat/lng），必要时引入 PostGIS。
- **地图实现**：Leaflet（开源）或 Google Maps（带 key 管理）。
- **聚合与筛选**：按时间/分类聚合。

交付物
- 摄影足迹地图可用。

## Phase 8：分享与移动端（Share Card / PWA / 手势）
- **分享卡片**：前端 Canvas 生成、含 EXIF/签名/二维码。
- **PWA**：manifest + service worker；离线缓存策略（thumb 优先）。
- **移动端手势**：左右滑切换、双击缩放、分享系统菜单。

交付物
- 可安装、可离线、可分享的移动端体验。

## Phase 9：语义搜索与智能相册（v1.x）
- **向量化**：为每张照片生成 embedding（后端异步任务）。
- **向量库**：pgvector（Postgres 内）优先，减少引入新基础设施。
- **智能相册规则**：SQL 规则 + UI 规则编辑器。

交付物
- “用自然语言找图”的核心差异化能力。

## 验收清单（每个阶段都适用）
- 功能：端到端可用（上传→列表→详情→互动→后台管理）。
- 权限：后端强制拦截（不靠前端隐藏）。
- 性能：首页首屏、滚动、详情切换有明确指标与回归。
- 成本：AI 与存储有监控与限额策略。
