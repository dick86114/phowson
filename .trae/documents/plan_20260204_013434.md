## 目标与边界
- 目标：把当前前端基于 localStorage 的 Mock 数据层替换为云端持久化（Supabase 底层 Postgres），保证现有前端页面（[Admin.tsx](file:///home/zl/work/workspaces/phowson/pages/Admin.tsx)、[Upload.tsx](file:///home/zl/work/workspaces/phowson/pages/Upload.tsx)、[PhotoDetail.tsx](file:///home/zl/work/workspaces/phowson/pages/PhotoDetail.tsx)）的 CRUD/点赞/评论/统计接口可用。
- 约束：浏览器无法直接连 Postgres；你只提供了 Postgres 连接串 `postgres:postgres@192.168.31.60:54321/phowson_gpt`，未提供 Supabase Project URL/Anon Key，因此采用“轻量后端 API + Postgres”方式接入（后续拿到 Supabase URL/Key 可再切换到 supabase-js/Row Level Security）。

## 数据库设计（Postgres）
- 新建表（建议）：
  - users：id、name、avatar、role、created_at
  - photos：id(UUID)、owner_user_id、title、description、category、tags(text[])、exif(jsonb)、image_url(text)、created_at、updated_at、views_count、likes_count
  - photo_likes：photo_id、user_id、created_at（联合唯一约束 photo_id+user_id，用于“点赞切换”）
  - photo_comments：id、photo_id、user_id、content、created_at
- 索引：photos(created_at)、photos(category)、photo_likes(photo_id)、photo_comments(photo_id)
- 兼容现有前端字段：
  - 继续返回 `viewsCount/likesCount/comments/likes` 这些字段结构，保持 [api.ts](file:///home/zl/work/workspaces/phowson/api.ts) 当前返回形状尽量不变，降低改动面。

## 数据迁移策略（localStorage → 云端）
- 两条路径都提供：
  1) 后端启动时可选“自动 Seed”：把 [constants.ts](file:///home/zl/work/workspaces/phowson/constants.ts) 的 `MOCK_PHOTOS` 导入数据库（只在空库或显式开关时执行）。
  2) 一次性导入本地数据：新增一个管理接口（例如 POST /admin/migrate/localstorage），接收前端上传的 localStorage photos 数组，后端做 upsert（按 id 或 title+created_at）避免重复。
- 导入完成后：前端切换到云端 API，localStorage 保留但不再作为数据源（可在设置里提供“清空本地缓存”按钮）。
