# 技术架构与开发规范 (Technical Architecture & Standards)

## 1. 系统架构设计

### 1.1 总体架构
Phowson 采用前后端分离架构，前端为单页应用 (SPA)，后端为 RESTful API 服务。

- **前端**: React + TypeScript + Vite + Tailwind CSS
- **后端**: Node.js + Fastify + PostgreSQL (pgvector)
- **AI 服务**: Google Gemini / OpenAI Compatible API
- **存储**: 本地文件系统 / S3 兼容对象存储

### 1.2 目录结构
```
phowson/
├── components/       # 前端组件库
├── pages/            # 前端页面路由
├── hooks/            # 自定义 React Hooks
├── server/           # 后端服务根目录
│   ├── routes/       # API 路由定义
│   ├── lib/          # 核心业务逻辑库 (AI, Auth, Image)
│   ├── db/           # 数据库访问层 (DAL)
│   ├── migrations/   # 数据库迁移脚本
│   └── plugins/      # Fastify 插件 (Auth, Error)
├── doc/              # 项目文档
└── docker/           # Docker 部署配置
```

### 1.3 核心模块划分
- **Auth Module**: 负责用户注册、登录、Session 管理与 RBAC 鉴权。
- **Photo Module**: 处理照片上传、元数据解析 (EXIF)、变体生成与存储。
- **AI Module**: 集成 LLM 能力，提供自动填单、语义搜索与摄影点评。
- **Gamification Module**: 实现 Streak、Badge、Leaderboard 等激励机制。

## 2. 数据库设计

### 2.1 核心表结构
- **users**: 用户表 (id, email, password_hash, role, avatar_url)
- **photos**: 照片表 (id, user_id, url, exif_data, ai_analysis, status)
- **albums**: 相册表 (id, name, query_rule)
- **activity_logs**: 活动日志表 (user_id, date, action_type, count)
- **badges**: 徽章定义表 (id, name, condition_rule)
- **user_badges**: 用户徽章关联表

### 2.2 关键技术点
- **JSONB 存储**: 大量使用 JSONB 存储非结构化数据 (如 EXIF、AI 分析结果)，保持 schema 灵活性。
- **Vector Search**: 使用 `pgvector` 存储图片/文本的 Embedding，支持语义搜索。
- **Spatial Index**: 使用 PostGIS (可选) 或经纬度索引支持地图查询。

## 3. 代码规范

### 3.1 命名规范
- **文件/目录**: 使用 `kebab-case` (如 `user-profile.tsx`, `api-routes.js`) 或 `PascalCase` (组件如 `UserProfile.tsx`)。
- **变量/函数**: 使用 `camelCase` (如 `getUserById`, `isValid`)。
- **常量**: 使用 `UPPER_SNAKE_CASE` (如 `MAX_UPLOAD_SIZE`)。
- **数据库**: 表名 `snake_case` 复数 (如 `users`), 字段名 `snake_case` (如 `created_at`)。

### 3.2 开发规范
- **类型安全**: 前后端均需定义明确的 TypeScript Interface/Type。
- **错误处理**: 后端统一抛出 `HttpError`，前端统一捕获并 Toast 提示。
- **注释**: 复杂逻辑必须写中文注释；导出的函数/组件需写 JSDoc。
- **Git Commit**: 遵循 Conventional Commits (如 `feat: add user login`, `fix: resolve upload error`)。

## 4. 关键算法说明

### 4.1 瀑布流布局算法
采用列高度贪心算法：维护 N 列的高度数组，新图片总是插入到当前高度最小的那一列，以保证底部尽量对齐。

### 4.2 AI 填单逻辑
1. 提取图片 EXIF 中的 GPS 与时间。
2. 调用逆地理编码 API 获取 `City, Country`。
3. 将图片缩略图传给 Gemini Vision 模型，Prompt: "分析图片内容，生成标题、描述与标签"。
4. 合并 EXIF 信息与 AI 生成结果，返回前端确认。

### 4.3 习惯养成 (Streak) 计算
基于 `activity_logs` 表，查询用户连续 N 天有 `upload` 或 `publish` 记录。若中断则重置，若通过补签卡 (Item) 可恢复。
