# 后续开发路线图（v1.0 → v1.x）

本文档基于 [产品需求文档 (PRD)_ Phowson - 浮生 (PhotoLogs) v1.0](./产品需求文档%20(PRD)_%20Phowson%20-%20浮生%20(PhotoLogs)%20v1.0.md) 与当前代码现状整理，目标是把后续迭代从“概念”落到“可执行”。

## 当前基线

- 数据层：已完成从 localStorage 到 Postgres 的云端持久化（Fastify + pg）。
- 功能链路：列表 / 详情 / 上传 / 后台 / 点赞评论 / 统计已打通。
- 已知短板：模拟登录、缺少后端强制 RBAC；图片对象存储未接入；AI Key 仍在前端环境变量。

## 迭代原则

- 优先顺序：安全与数据边界（Auth/RBAC/存储） > 性能体验（瀑布流/虚拟列表） > 增长玩法（Streak/Heatmap/分享/PWA） > 差异化能力（语义搜索/智能相册）。
- 任何阶段都必须提供：最小可回归验证清单、日志与错误码、可回滚策略。

## Phase 0：稳定化与工程化（建议先做）

目标：把“能跑”升级为“可维护、可观测、可测试”。

- 后端模块化：路由拆分、统一错误格式、请求校验、统一鉴权入口。
- 指标与日志：请求链路 ID、慢查询监控、错误报警（先本地可见）。
- 自动化测试：API 冒烟测试覆盖核心链路。

## Phase 1：身份认证与 RBAC（从 UI 限制到 API 强制）

目标：实现真实登录与后端强制权限控制，保证数据边界。

- 认证方案：Clerk（推荐）或自建 JWT（邮箱+密码+刷新 token）。
- RBAC：Admin/Family/Guest 权限矩阵在后端落地（禁止靠前端隐藏）。
- 用户/分类管理：把当前“前端临时状态”改为后端真实表。

## Phase 2：对象存储与图片管线（性能与成本的关键）

目标：原图与缩略图走对象存储 + CDN，数据库仅保存元数据与变体信息。

- 方案：S3/R2/MinIO 等 S3 兼容对象存储。
- 变体：thumb/medium/original，多尺寸按需加载。
- 安全：上传鉴权、MIME 白名单、大小限制、图片处理任务。

## Phase 3：画廊体验升级（瀑布流 + 高性能加载）

目标：沉浸式与高性能并行，支撑千级照片。

- Masonry：瀑布流布局（库或自研），支持不同比例照片。
- 虚拟化：长列表滚动不卡顿。
- 体验：Skeleton/BlurHash/LQIP，详情页预加载上一张/下一张。

## Phase 4：上传工作流增强（批量 + EXIF 完整）

目标：摄影师高效整理，EXIF 100% 准确。

- 批量上传：队列、并发、失败重试。
- EXIF 归一：前端读取 + 后端二次校验，统一字段写入 JSONB。
- 表单：自动建议分类/标签，批量应用策略。

## Phase 5：AI 增强（填单 + 摄影导师）

目标：AI 真正融入工作流且成本可控。

- Prompt 分层：标题/描述/标签（通用）；AI 点评（仅 Admin 可见）。
- 成本：缓存、批量合并、限流、失败降级。
- 安全：把 Gemini 调用移出前端（后端或 serverless）。

## Phase 6：习惯养成系统（Streak / Heatmap / 成就）

目标：从展示作品升级为“习惯养成陪伴”。

- activity_logs：每日上传/发布活动记录。
- Heatmap：过去一年贡献图与筛选。
- 徽章/挑战：规则引擎与定时任务（先简单后扩展）。

## Phase 7：地图模式（摄影足迹）

目标：基于 GPS 数据生成“足迹地图”。

- GPS：EXIF 读取并规范化，缺省策略明确。
- 地图：Leaflet 或 Google Maps；聚合与筛选。
- 可选：PostGIS / 地理索引（规模上来再引入）。

## Phase 8：分享与移动端（Share Card / PWA / 手势）

目标：适配移动端传播与离线使用。

- 分享卡片：Canvas 生成（照片+EXIF+签名+二维码）。
- PWA：manifest + SW 缓存策略（thumb 优先）。
- 手势：左右滑切图、双击缩放、系统分享。

## Phase 9：语义搜索与智能相册（v1.x 差异化）

目标：用自然语言找图与规则相册，构建“第二大脑”特性。

- 向量化：embedding 异步生成与更新。
- 向量库：优先 pgvector，减少新基础设施。
- 智能相册：规则编辑器（SQL/DSL）+ 保存视图。
