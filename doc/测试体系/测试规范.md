# 测试规范 (Test Strategy & Specification)

## 1. 测试流程

### 1.1 单元测试 (Unit Test)
- **范围**: 核心业务逻辑 (AI 解析、EXIF 读取、工具函数)。
- **工具**: Node.js 原生 Test Runner / Vitest。
- **通过标准**: 代码覆盖率 > 80%，所有 Case Pass。

### 1.2 集成测试 (Integration Test)
- **范围**: API 接口测试 (Database <-> API)。
- **工具**: Supertest + Docker (Test DB)。
- **通过标准**: 核心链路 (上传、登录、搜索) 正常，无 5xx 错误。

### 1.3 端到端测试 (E2E Test)
- **范围**: 用户真实操作流程模拟。
- **工具**: Playwright / Cypress。
- **通过标准**: 关键 User Journey (从登录到发布) 成功闭环。

### 1.4 手工测试 (Manual Test)
- **范围**: UI/UX 体验、异常边界、兼容性测试。
- **执行人**: QA / PM。

## 2. 缺陷管理流程

1. **发现 Bug**: 测试人员在 Issue Tracker (GitHub/Jira) 提交 Bug。
2. **定级**:
   - **P0 (Critical)**: 系统崩溃、数据丢失、核心功能不可用。
   - **P1 (Major)**: 主要功能受损，无临时方案。
   - **P2 (Minor)**: 次要功能缺陷，UI 问题。
3. **修复**: 开发人员认领并修复 Bug，提交 PR。
4. **验证**: 测试人员在测试环境回归验证。
5. **关闭**: 验证通过后关闭 Issue。

## 3. 测试环境配置

- **Dev**: 开发本地环境，使用 Mock 数据。
- **Test**: 测试服务器，每日自动部署 master 分支，使用测试数据库 (定期清洗脱敏)。
- **Prod**: 生产环境，仅允许 Smoke Test。

## 4. 自动化测试接入 CI

- **Pre-commit**: 运行 Lint 和 Unit Test。
- **PR Check**: 运行 Integration Test，阻断失败合并。
- **Nightly Build**: 运行全量 E2E 测试。
