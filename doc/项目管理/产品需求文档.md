## 1. 项目概述 (Project Overview)

### 1.1 产品愿景

打造一个“摄影师的第二大脑与习惯养成伴侣”。

不同于 Instagram 的快餐式社交，Phowson 专注于：

- ​**高保真画质与沉浸体验**：无算法干扰，还原照片本质。
- ​**数据化摄影**：深度整合 EXIF 数据，辅助摄影师复盘技术参数。
- **习惯养成 (Streak)**  ：通过每日打卡机制，鼓励摄影师保持创作惯性。
- ​**AI 辅助工作流**：利用 Gemini 大模型简化整理工作，提供创作灵感。

### 1.2 核心用户画像

- **管理员 (Admin/Photographer)**  ：拥有所有权，负责上传、管理、查看深度数据分析。
- **家庭成员 (Family)**  ：拥有查看私密相册、发表评论的高级权限。
- **访客 (Guest)**  ：仅能查看公开作品，点赞互动。

## 2. 现有功能优化 (Optimizations)

基于当前代码库 (react + tailwind + genai) 的分析，以下是针对现有模块的优化需求：

### 2.1 画廊体验升级 (Gallery & Viewer)

​**现状**​：基础网格布局，全屏查看支持手势缩放。  
​**优化需求**：

- **瀑布流布局 (Masonry Layout)**

  - ​**问题**：目前 grid 布局在不同比例照片混合时会有空白或裁剪。
  - ​**方案**：引入瀑布流算法，完美展示竖构图和横构图照片，最大化屏幕利用率。
- **地图模式 (Map View)**

  - ​**现状**：exif.location 仅为文本。
  - ​**方案**：集成 Google Maps 或 Leaflet，将照片聚合在地图上，形成“摄影足迹”。
- **高性能加载**

  - ​**方案**：引入 BlurHash 或骨架屏 (Skeleton) 占位，优化图片加载时的视觉跳动；实现虚拟列表 (Virtualization) 以支持数千张照片的流畅滚动。

### 2.2 上传与 AI 增强 (Upload Workflow)

​**现状**​：单张上传，支持 Gemini 识别内容并填单。  
​**优化需求**：

- **真实的 EXIF 读取**

  - ​**问题**：目前 EXIF 需 AI 猜测或手动输入。
  - ​**方案**​：集成 `exifr`​ 或 `exif-js`​ 库，上传时 **100% 准确读取** 本地文件的元数据（光圈、快门、GPS、拍摄时间），AI 仅用于生成标题和描述。
- **批量上传 (Batch Upload)**

  - ​**方案**：支持一次性拖入多张照片，建立上传队列，批量应用 AI 分析。
- **AI 摄影导师 (AI Critique)**

  - ​**扩展**​：在 AI 分析时，增加一个 prompt 让 Gemini 扮演“资深摄影评论家”，对照片构图、光影给出 ​**评分和改进建议**，并保存为“AI 点评”字段，仅管理员可见。

### 2.3 移动端适配 (Mobile Experience)

​**现状**​：Tailwind 响应式支持良好，但交互偏桌面。  
​**优化需求**：

- ​**PWA 支持**：配置 manifest，使其能安装在手机桌面，支持离线查看缓存照片。
- ​**手势导航**：在照片详情页支持左右滑动切换上一张/下一张（目前仅支持点击返回）。

## 3. 新功能扩展 (New Features Expansion)

### 3.1 习惯养成系统 (Gamification & Streak)

​**背景**​：首页目前的 Streak Widget 仅为静态展示。  
​**功能描述**：

- **热力图 (Heatmap)**  ：类似 GitHub 的贡献图，展示过去一年的拍摄/发布频率。
- **成就徽章**

  - “夜猫子”：深夜上传照片超过 10 次。
  - “风景猎人”：发布 50 张 Landscape 分类照片。
  - “全勤奖”：连续打卡 30 天。
- **周挑战 (Weekly Challenge)**  ：系统每周自动生成一个 AI 命题（如“寻找红色的物体”），用户上传并在首页置顶展示。

### 3.2 智能相册 (Smart Albums)

​**背景**​：目前仅有静态分类。  
​**功能描述**：

- ​**动态筛选**：允许创建基于规则的相册，例如“所有 2023 年在 瑞士 拍摄的 使用 35mm 镜头 的照片”。
- **AI 语义搜索 (Semantic Search)**

  - 利用 Gemini 的多模态能力，支持自然语言搜索，如“搜索一张即使没有标记‘狗’，但画面里有狗在草地上奔跑的照片”。

### 3.3 社交与分享 (Social & Sharing)

​**功能描述**：

- ​**生成分享卡片**：点击分享时，动态生成一张包含照片、EXIF 参数、摄影师签名和二维码的精美海报图片（适合发朋友圈/Instagram）。
- ​**访客留言板**：除了单张照片评论，增加全站留言板功能。

## 4. 技术架构演进 (Technical Roadmap)

为了支撑上述功能，后端和数据层需要从当前的纯前端 Mock 模式向全栈架构迁移：

### 4.1 数据层迁移 (Data Layer Migration)

- ​**现状**​：使用 constants.ts 中的静态数组 (`MOCK_PHOTOS`)。
- ​**规划**：

  - ​**数据库**​：迁移至 **PostgreSQL**（自建或托管均可）。

    - ​`Photos` 表：存储图片元数据、EXIF、AI 分析结果。
    - ​`Users` 表：存储用户配置、权限角色。
    - ​`Streaks` 表：记录用户每日打卡活动日志。
  - **对象存储 (Object Storage)**

    - ​**现状**：依赖外部链接或 Base64（性能差）。
    - ​**规划**​：接入 **AWS S3** 或 **Cloudflare R2** 存储原图，并配置 **CDN** (如 Cloudinary) 进行动态裁剪和格式转换 (WebP/AVIF)，提升加载速度。

### 4.2 身份认证 (Authentication)

- ​**现状**​：前端模拟登录 (`localStorage` 存储 Token)。
- ​**规划**：

  - 集成 **Clerk** 或 ​**NextAuth.js**。
  - 实现真正的基于角色的访问控制 (RBAC)，确保 API 层面拦截非管理员的写入请求。

### 4.3 前端架构 (Frontend Architecture)

- ​**状态管理**​：引入 **TanStack Query (React Query)**  管理服务端状态，替代目前的 `useEffect`​ + `useState` 手动获取数据，解决缓存和同步问题。
- ​**路由**​：目前的 React Router 7 配置良好，后续可考虑迁移至 **Next.js** (App Router) 以获得更好的 SEO（服务端渲染 SSR）和图片优化组件。

## 5. UI/UX 设计规范 (Design Guidelines)

### 5.1 色彩系统 (Color System)

- ​**Primary (品牌色)** ​: `#137fec` (科技蓝) - 用于强调、按钮、激活状态。
- ​**Surface (界面底色)** :

  - ​`Light`​: `#f6f7f8`​ (背景), `#ffffff` (卡片)
  - ​`Dark`​: `#111a22`​ (背景 - 深蓝灰), `#1a2632` (卡片 - 提升层级)
- ​**Semantic (语义色)** :

  - ​`Error`​: `#ef4444` (红色)
  - ​`Success`​: `#22c55e` (绿色)
  - ​`Streak/Hot`​: `#f59e0b` (琥珀色 - 用于打卡火焰图标)

### 5.2 字体排印 (Typography)

- ​**字体栈**: Inter, 系统默认无衬线字体。
- ​**层级**:

  - ​**H1/Display**: 用于着陆页大标题，字重 900/700。
  - ​**Body**: 易读性为主，字重 400，行高 1.6。
  - ​**Meta**​: EXIF 数据和辅助信息，字重 500，字号 `12px` (xs)。

### 5.3 交互原则 (Interaction Principles)

- **微交互 (Micro-interactions)**  ：点赞时的心形跳动、Hover 图片时的遮罩渐变、页面切换的淡入淡出 (Fade In)。
- **无干扰模式 (Distraction Free)**  ：进入“全屏查看”或“上传”流程时，隐藏顶部导航栏和底部 Footer，让用户聚焦内容。

## 5. 需求变更管理流程 (Requirement Change Management)

为确保项目目标可控，所有需求变更需遵循以下流程：

1.  **提出变更**：任何项目成员均可提交《需求变更申请单》，说明变更原因、内容及预期价值。
2.  **初步评估**：产品经理 (PM) 评估变更的必要性与优先级。
3.  **影响分析**：
    *   技术负责人评估对架构、代码、性能的影响。
    *   测试负责人评估对测试用例和回归测试的影响。
    *   项目经理评估对进度和资源的影响。
4.  **变更评审**：召开变更评审会议 (CCB)，决策是否批准变更。
5.  **执行变更**：
    *   更新 PRD 及相关设计文档。
    *   更新 Backlog 和 Roadmap。
    *   通知所有相关人员。
6.  **验收关闭**：变更开发完成后，需进行专项验收，通过后关闭变更单。
